## Major Findings

1. The evaluated policy is not the policy that is actually trained for the online task. The offline policy trainer in [scripts/train_intervention_policy.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_intervention_policy.py#L140) synthesizes detector confidence from ground-truth hallucination density and uses a hand-written reward in [scripts/train_intervention_policy.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_intervention_policy.py#L172). It never loads the detector or the base model, and the production pipeline evaluates `checkpoints/intervention_policy/best_policy.pt` in [scripts/run_production.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_production.sh#L165) after training that offline proxy in [scripts/run_production.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_production.sh#L118). There is an online trainer in [scripts/train_policy_online.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_policy_online.py#L1), but Stage 4 never consumes its output. Minimum fix: make the online policy the default evaluated artifact, or regenerate an offline dataset with real detector outputs and real intervention outcomes instead of synthetic rewards.

2. The evaluation code has two experiment-invalidating issues. First, the advertised oracle baseline is not oracle: [scripts/eval_chi.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/eval_chi.py#L322) backtracks at the detector-predicted onset returned by `generate_with_onset_detection`, not at ground-truth onset. Second, intervention methods decode the full prompt plus answer in [scripts/eval_chi.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/eval_chi.py#L303), [scripts/eval_chi.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/eval_chi.py#L339), and [scripts/eval_chi.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/eval_chi.py#L384), while non-intervention baselines decode answer-only text in [scripts/eval_chi.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/eval_chi.py#L249). That makes token counts, perplexity, and some factuality comparisons inconsistent across methods. Minimum fix: use labeled traces for true-onset oracle evaluation and always decode/evaluate only the suffix after `prompt_len`.

3. Multi-GPU support is mostly declarative, not implemented. The pipeline advertises multi-GPU support and even computes a `torchrun` command in [scripts/run_all_experiments.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_all_experiments.sh#L27), but every training stage still launches plain `python` in [scripts/run_all_experiments.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_all_experiments.sh#L102), [scripts/run_all_experiments.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_all_experiments.sh#L145), and [scripts/run_all_experiments.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_all_experiments.sh#L172). The training scripts only read `LOCAL_RANK` in [scripts/train_onset_detector.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_onset_detector.py#L437) and [scripts/train_intervention_policy.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_intervention_policy.py#L312); they never initialize distributed training, never shard data, and never gate checkpoint writes by rank. If launched under `torchrun`, every rank would duplicate the full job and contend for the same checkpoint files. Minimum fix: either implement actual DDP/Accelerate with `DistributedSampler`, rank-safe checkpointing, and rank-0-only writes, or explicitly downgrade the claim to single-process model sharding only.

4. Checkpoint resume support exists only at a coarse level and is not reproducible. The phase-marker logic in [scripts/run_all_experiments.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_all_experiments.sh#L33) and [scripts/run_production.sh](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/run_production.sh#L61) can skip completed stages, but Stage 1 trace collection keeps all traces in memory and only writes outputs at the end of a dataset in [scripts/collect_traces.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/collect_traces.py#L350) and [scripts/collect_traces.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/collect_traces.py#L411), so an interruption loses all in-progress work. The explicit `--resume_from_checkpoint` argument is also misleading: both training scripts ignore the provided path and just pick the latest file from the output directory in [scripts/train_onset_detector.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_onset_detector.py#L235) and [scripts/train_intervention_policy.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_intervention_policy.py#L348). More importantly, detector training recreates a fresh `random_split` before loading a checkpoint in [scripts/train_onset_detector.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_onset_detector.py#L214) and [scripts/train_onset_detector.py](/home/nwh/nips_15/github_repos/nips-trace-hallu/scripts/train_onset_detector.py#L327), so a resumed run may continue on a different train/validation partition. Minimum fix: persist split indices and RNG state, honor explicit checkpoint paths, and checkpoint Stage 1 incrementally.

Minimum fix: remove or repair legacy paths, align README claims with the runnable code, and add an end-to-end smoke test plus expected artifact manifest.

## Requested Focus Areas

Code quality and completeness: the codebase is reasonably modular and syntactically valid, but there is too much drift between README claims, pipeline scripts, and what the core experiment code actually does.

Multi-GPU support: not sufficient for reviewer-grade reproducibility. Current support is limited to model sharding via `device_map="auto"` and embarrassingly parallel ablations, not safe distributed training.

Checkpoint resume support: partial. Phase skipping works, but fine-grained recovery is weak and resumed detector training is not guaranteed to continue the same experiment.

Readiness to produce experimental results: no. The current code is still a research prototype; the offline/online policy mismatch and evaluation bugs are blockers for trusting reported numbers.

## Actionable Next Steps

1. Make the online policy training path the default path used by evaluation, or remove policy-learning claims until the offline trainer is made faithful to the deployed state/action/reward distribution.
2. Fix `eval_chi.py` so every method evaluates answer-only text, and implement a real oracle baseline based on labeled onset positions.
3. Either implement true distributed execution end to end or remove all multi-GPU training claims from the README and runner scripts.
4. Persist dataset splits, RNG state, and per-stage progress so checkpoint resume is deterministic and interruption-safe.
5. Add a tiny, one-command smoke test that runs trace collection, detector training, policy training, and evaluation on a handful of samples and verifies artifact creation.
